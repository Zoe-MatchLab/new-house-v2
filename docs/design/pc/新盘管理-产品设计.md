# 新盘管理 - 产品设计文档

> 版本：1.5
> 日期：2026-04-13
> 端：PC/Web端

---

## 1. 概述

### 1.1 设计目标

为新房项目维护人员提供便捷的新盘信息录入、审核、上线管理功能。

### 1.2 设计原则

- **清晰高效**：减少操作步骤，快速完成录入
- **信息完整**：必填校验确保数据质量
- **状态明确**：各状态用颜色区分，一目了然

### 1.3 页面清单

| 页面 | 功能 | 文件 |
|-----|------|-----|
| 新盘列表 | 分页展示、筛选、批量操作 | new-house-list.html |
| 新建楼盘 | 81字段表单录入 | new-house-edit.html |
| 编辑楼盘 | 修改楼盘信息 | new-house-edit.html?id={id} |
| 楼盘详情 | 完整展示楼盘信息 | new-house-detail.html |
| 上线配置 | 配置可下发人员、通知人 | new-house-launch.html |
| 审核页面 | 审核通过/驳回 | new-house-detail.html（Tab） |

## 7. 业务规则

### 7.1 状态操作规则

| 规则 | 说明 |
|-----|------|
| 提交审核后不可修改 | 提交审核后，楼盘信息锁定，只能审核或撤回 |
| 撤回操作 | 待审核状态可撤回，撤回后变为草稿状态，可重新编辑 |
| 审核通过后不可删除 | 审核通过后的楼盘不允许删除，确保数据完整性 |
| 下线后任务终止 | 下线操作会终止关联的进行中任务 |
| 删除直接删除 | **不设置回收站**，删除操作直接永久删除 |

### 7.1.1 新建/编辑页面规则

| 规则 | 说明 |
|-----|------|
| 页面复用 | 新建与编辑共用同一页面，通过URL参数区分 |
| 标题显示 | 新建："新建楼盘"；编辑："编辑楼盘-{状态}" |
| 状态提示 | 编辑页顶部显示当前楼盘状态标签 |
| 必填校验 | 提交审核时校验，未填写高亮缺失字段 |
| 保存方式 | 草稿保存（不校验）/ 提交审核（校验必填） |

### 7.1.2 楼盘详情页规则

| 规则 | 说明 |
|-----|------|
| 按钮逻辑 | 根据状态显示不同操作按钮（见权限矩阵） |
| 撤回入口 | 待审核状态显示"撤回"按钮 |
| 删除限制 | 审核通过后（包括已上线、已下线）不可删除，仅草稿/已驳回状态可删除 |
| 编辑跳转 | 点击"编辑"需校验权限 |

### 7.2 操作权限矩阵

| 操作 | 草稿 | 待审核 | 审核通过 | 已驳回 | 已上线 | 已下线 |
|-----|-----|-------|---------|-------|-------|-------|
| 详情 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 编辑 | ✅ | ❌ | ⚠️✅(超管) | ✅ | ⚠️✅(超管) | ✅ |
| 提交审核 | ✅ | — | — | ✅ | — | — |
| 撤回 | — | ✅ | — | — | — | — |
| 审核 | — | ✅ | — | — | — | — |
| 配置 | — | — | ✅ | — | — | ✅ |
| 上线 | — | — | ✅ | — | — | ✅ |
| 下线 | — | — | — | — | ✅ | — |
| 删除 | ✅ | ❌ | ❌(置灰) | ❌(置灰) | — | ❌(置灰) |

## 8. 下线确认弹窗

### 8.1 弹窗结构

```
┌─────────────────────────────────────────┐
│  ⚠️ 确认下线                              │
├─────────────────────────────────────────┤
│  下线后，该楼盘关联的探盘任务将被终止：      │
│                                          │
│  ┌─────────────────────────────────────┐ │
│  │ 进行中的任务：   3个将终止 (红色)      │ │
│  │ 已分配经纪人：   5人将收到通知         │ │
│  └─────────────────────────────────────┘ │
│                                          │
│           [取消]  [确认下线]               │
└─────────────────────────────────────────┘
```

### 8.2 数据来源（实时API）⚠️

| 字段 | 数据来源 | 说明 |
|-----|---------|------|
| 进行中的任务数 | 后端API实时查询 | GET /api/buildings/{id}/pending-tasks |
| 已分配经纪人数 | 后端API实时查询 | GET /api/buildings/{id}/assigned-brokers |

> ⚠️ **必须实时调用API**：不可使用静态数据，必须在弹窗显示前调用后端接口获取最新关联任务数。

### 8.3 交互逻辑

1. 点击"下线"按钮
2. **调用API获取关联任务数**（显示loading态）
3. 展示确认弹窗（包含关联任务信息）
4. 用户确认后调用下线API
5. 发送下线通知给区域经理+经纪人
6. 终止关联任务
7. 刷新列表
8. 区域经理小程序侧不再显示该楼盘

### 8.4 通知触发规则 ⭐重要

| 触发时机 | 通知对象 | 通知内容 |
|---------|---------|---------|
| 新盘上线 | 区域经理 | "新楼盘'{name}'已上线，请配置探盘任务" |
| 楼盘下线 | 区域经理+进行中经纪人 | "楼盘'{name}'已下线，关联任务已终止" |

> **核心规则**：通知是区域经理/经纪人能够看到楼盘的前提条件。未收到通知前，小程序侧不显示该楼盘。
